# 🎯 实验自动执行规范（AGENTS.md）

## 🌏 语言与输出规范（强约束）
1. 所有思考、分析、解释必须使用中文
2. 所有代码注释必须使用中文
3. 所有实验报告必须使用中文
4. 输出必须结构清晰、分点说明、逻辑严谨

---

## 🎯 核心执行原则
1. 先读文档，再写代码（强制）
2. 严格按实验要求实现，不得自行发挥
3. 最小修改原则（只改必要代码）
4. 严格限制修改范围
5. 每个实验必须生成报告 + git 版本
6. 所有实验必须可回溯（版本化管理）
7. 本规范应优先保证“可复用性”，禁止把具体课程、具体实验名、具体路径硬编码进执行逻辑
8. 若编译、测试、运行环境缺失或未正确配置，必须先处理环境问题，再进入代码实现阶段
9. 每次运行成功后以及通过测试集后，必须保留输出结果截图，并写入实验报告

---

## 📂 适用范围
本规范适用于“按实验说明完成代码修改、生成实验报告、进行 git 版本管理”的开发任务。

可适配：
- 操作系统实验
- 编译原理实验
- 数据库实验
- 计算机网络实验
- 其他具有“实验文档 + 代码目录 + 报告输出 + git版本管理”结构的课程实验

---

## ⚙️ 启动前信息收集机制（必须先完成）

当用户第一次要求开始某个实验时，如果以下信息不完整，必须先主动询问；不得在信息缺失时直接修改代码。

### 必须收集的信息
1. 实验名称
2. 实验代码目录
3. 实验文档路径
4. 报告输出路径
5. 允许修改的文件或目录范围
6. git 仓库根目录
7. 是否存在实验继承关系图或参考资料
8. 当前实验是否有固定命名规则（如 lab1_challenge2）
9. 编译、测试、运行方式
10. 是否已有可用环境，或是否允许在项目目录下补充配置环境

### 推荐询问模板
请提供以下信息后再开始实验：
1. 实验名称：
2. 实验代码目录：
3. 实验文档路径：
4. 报告输出路径（若不提供，则默认生成在当前实验目录下）：
5. 允许修改的文件/目录范围：
6. git 仓库根目录：
7. 是否有继承关系图、指导文件夹或补充资料？路径是什么？
8. 是否有版本命名规则？若没有，将使用默认规则。
9. 该实验的编译命令、测试命令、运行命令分别是什么？
10. 如果环境缺失，是否允许我在项目目录下自行补充配置？若不允许，请告诉我由你负责配置。

### 默认约定（用户未指定时可建议）
若用户不想逐项提供，可建议采用以下统一结构：
- 指导资料统一放入：`指导/`
- 实验文档统一放入：`指导/实验文档/`
- 继承关系图统一放入：`指导/`
- 代码统一放入：`代码/`
- 每个实验报告默认放在对应实验目录下
- 运行截图与测试截图默认放在对应实验目录下的 `report_assets/` 目录中

## 📚 指导资料约定（推荐）
为提高复用性和自动执行稳定性，优先建议用户将资料整理为以下结构：

```text
项目根目录/
├── 指导/
│   ├── 实验文档/
│   ├── 继承关系图.png
│   └── 其他参考资料...
├── 代码/
│   ├── 实验A/
│   ├── 实验B/
│   └── ...
├── report_assets/
└── AGENTS.md
```

若用户未采用该结构，也必须以“用户实际提供的路径”为准，不得自行假设。

## 🧩 实验配置记录规则

每次开始一个新实验前，必须先整理并确认当前实验配置。

### 当前实验配置应至少包含

- 实验名称：`<experiment_name>`
- 实验目录：`<experiment_dir>`
- 实验文档：`<experiment_doc>`
- 报告路径：`<report_path>`
- 允许修改范围：`<allowed_paths>`
- 仓库根目录：`<repo_root>`
- 补充资料路径：`<extra_reference_paths>`
- 版本命名规则：`<version_rule>`
- 编译命令：`<build_command>`
- 测试命令：`<test_command>`
- 运行命令：`<run_command>`
- 截图资源目录：`<report_assets_dir>`
- 环境配置状态：`<environment_status>`
- 是否允许在项目目录内自行补充环境：`<environment_permission>`

在配置不清晰前，不得直接进入代码修改阶段。

## 🛠️ 环境检查与配置规则（必须执行）

### 环境检查触发条件

在开始编码前，必须先检查以下内容是否可用：

1. 编译工具链是否可用
2. 测试命令是否可执行
3. 运行命令是否可执行
4. 依赖是否齐全
5. 项目要求的容器、虚拟环境、解释器、编译器、模拟器、构建工具是否齐全
6. 当前目录结构是否满足实验运行要求

### 当环境未配置好时的处理规则

若发现编译、测试、运行环境未配置好，必须暂停代码实现，先输出以下内容并向用户确认：

1. 当前缺失的环境项
2. 每个环境项的用途
3. 建议的配置方式
4. 是否需要在项目目录下新增配置文件、脚本、依赖目录或容器配置
5. 这些配置会修改哪些文件
6. 这些修改是否只用于环境搭建而非实验逻辑实现

### 必须向用户确认的标准话术要求

当环境缺失时，必须明确询问用户：

当前编译/测试/运行环境未配置完成。
是否允许我在项目目录下自行补充环境配置？

我计划配置的内容包括：

- 缺失环境 1：用途说明
- 缺失环境 2：用途说明
- 缺失环境 3：用途说明

预计会新增或修改以下内容：

- 文件/目录 A
- 文件/目录 B
- 文件/目录 C

这些改动仅用于让项目可以正常编译、测试和运行，不涉及实验业务逻辑修改。
请确认是否允许我执行。

### 若允许自行配置

若用户明确允许，则可以在项目目录下补充必要环境配置，但必须遵守：

1. 优先使用项目内局部配置，不污染全局环境
2. 优先新增独立配置文件，避免破坏原项目
3. 所有环境相关新增内容必须可说明、可回退、可追踪
4. 环境配置也必须写入实验报告
5. 环境配置完成后，必须重新验证编译、测试、运行流程

### 若不允许自行配置

若用户不允许，则必须：

1. 明确指出缺失项
2. 告知用户需要其协助完成哪些操作
3. 为用户提供可直接执行的详细步骤
4. 在用户完成前，不得继续代码实现

### 需要用户协助时的输出要求

若需要用户配置环境，必须明确写出：

1. 要安装什么
2. 去哪里安装
3. 在哪个目录执行
4. 执行哪些命令
5. 配置完成后如何验证
6. 验证成功后需要把什么输出反馈回来

### 环境配置说明必须包含的内容

若执行或建议环境配置，必须说明：

1. 环境名称
2. 作用
3. 为什么当前实验需要它
4. 安装或配置位置
5. 对项目的影响范围
6. 是否需要用户权限、网络、系统依赖或容器支持

------

## 🚫 代码修改限制（强约束）

只能修改“用户明确允许修改”的文件或目录。

### 执行规则

1. 若用户已明确给出允许修改范围，则严格以用户要求为准
2. 若用户未给出允许修改范围，必须先询问
3. 禁止修改实验目录之外的无关文件
4. 禁止擅自扩大修改范围
5. 若发现实现目标必须修改范围外文件，必须先向用户说明并征求确认
6. 环境配置文件与实验逻辑文件应区分说明，禁止将环境改动混入业务改动而不告知用户

------

## 🔄 实验执行流程（统一）

当用户输入：

- 开始某个实验
- 实现某个实验
- 修改某个实验
- 继续某个实验

必须执行以下流程：

### 第一步：确认实验配置

1. 收集或确认实验名称
2. 定位实验目录
3. 定位实验文档
4. 确认允许修改范围
5. 确认报告输出路径
6. 确认 git 仓库根目录
7. 确认编译、测试、运行命令
8. 确认截图保存目录
9. 确认环境配置权限

### 第二步：检查环境

1. 检查编译环境
2. 检查测试环境
3. 检查运行环境
4. 若环境缺失，则先向用户确认是否允许在项目目录下自行配置
5. 若需要用户协助，则输出详细操作步骤并等待用户完成
6. 环境就绪后，再继续后续流程

### 第三步：阅读文档

1. 阅读实验文档
2. 提炼实验目标
3. 总结实验要求
4. 若存在继承关系图、参考代码或补充资料，也应一并阅读

### 第四步：版本初始化检查

1. 检查是否已存在该实验的 initial 版本
2. 若不存在，则先保存 initial
3. 若已存在，则跳过

### 第五步：代码分析

1. 分析需要修改的文件
2. 分析需要修改的函数
3. 给出修改思路
4. 在未分析清楚前，不得直接改代码

### 第六步：实现代码

1. 仅在允许范围内修改
2. 保持最小改动
3. 保持原有代码风格一致
4. 关键逻辑补充中文注释

### 第七步：编译、运行与测试

1. 执行编译
2. 执行运行
3. 执行测试
4. 每次运行成功后保存输出截图
5. 通过测试集后保存输出截图
6. 整理输出结果，用于写入实验报告

### 第八步：生成实验报告

1. 按规范输出实验报告
2. 插入运行成功截图
3. 插入测试通过截图
4. 保存到约定路径

### 第九步：保存版本

1. 自动递增版本号
2. 提交 git commit
3. 创建对应 tag

## 🧪 initial 版本规则（必须执行）

### initial 命名规则

若用户未提供规则，则默认采用：

```bash
git add -A
git commit -m "chore: save <experiment_name> initial"
git tag <experiment_name>_initial
```

### 规则说明

1. 每个实验只应创建一次 initial
2. 若 `<experiment_name>_initial` 已存在，则跳过
3. 禁止重复覆盖 initial

------

## 📸 运行与测试截图规则（必须执行）

### 截图触发时机

以下场景必须截图：

1. 程序成功运行后
2. 测试集全部通过后
3. 若实验要求中有关键中间结果，也应截图保留
4. 若环境配置成功并完成首次有效验证，也建议截图保留

### 截图保存规则

若用户未指定截图目录，则默认保存到：

```
<experiment_dir>/report_assets/
```

若目录不存在，应先创建。

### 截图命名规则

默认命名如下：

- `run_success_01.png`
- `run_success_02.png`
- `test_pass_01.png`
- `test_pass_02.png`
- `env_check_success_01.png`

若用户有命名规则，则以用户要求为准。

### 截图内容要求

截图必须尽量完整保留以下信息：

1. 执行命令
2. 关键输出结果
3. 成功标志、通过标志或评分结果
4. 能证明“运行成功”或“测试通过”的核心证据

### 截图写入报告规则

实验报告中必须单独设置“运行结果截图”与“测试结果截图”章节，写明：

1. 截图文件名
2. 截图对应的命令
3. 截图说明
4. 运行或测试结果结论

## 🏷️ 版本保存规则（自动递增）

### 默认版本命名

若用户未提供规则，则默认：

```bash
git add -A
git commit -m "feat: complete <experiment_name> vN"
git tag <experiment_name>_vN
```

### 版本号获取规则

1. 使用 git tag 查询当前实验已有版本：

```bash
git tag | grep <experiment_name>_v
```

1. 提取最大版本号
2. 在其基础上 +1
3. 禁止覆盖已有版本

## 🔁 版本切换机制

当用户输入：

- 切换到某个版本
- 回到某个版本
- checkout 到某个版本

执行：

```bash
git checkout <target_version>
```

### 必须输出

1. 目标版本
2. 该版本是否存在
3. 是否切换成功
4. 当前版本
5. 是否处于 detached HEAD

### 若用户要继续开发，必须提示

当前处于 detached HEAD 状态，
如需继续开发，建议新建分支后再修改，例如：

```bash
git checkout -b <target_version>_dev
```

## ✅ 完成判定标准

一个实验完成必须满足：

- ✔ 已确认实验配置
- ✔ 已检查环境
- ✔ 若环境缺失，已完成确认或用户协助配置
- ✔ 已阅读文档
- ✔ 使用正确目录
- ✔ 未越界修改
- ✔ 已完成编译
- ✔ 已成功运行
- ✔ 已通过测试（若实验提供测试集）
- ✔ 已保存运行成功截图
- ✔ 已保存测试通过截图
- ✔ 已生成报告
- ✔ 已保存 initial（首次）
- ✔ 已保存版本（vN）

否则视为未完成。

## 🚫 禁止行为

1. 不读文档直接写代码
2. 在实验信息不完整时擅自开始
3. 环境未确认前直接硬改代码
4. 环境缺失却不告知用户
5. 未经确认擅自在项目外污染全局环境
6. 修改非允许文件
7. 在错误目录修改
8. 不生成报告
9. 不进行 git 保存
10. 不保留运行结果与测试结果截图
11. 覆盖历史版本
12. 将特定课程路径、实验名、目录结构写死到通用规则中

## 💡 执行优化规则

1. 优先最小修改
2. 保持代码风格一致
3. 关键逻辑必须写中文注释
4. 报告必须详细完整
5. 优先复用已有配置，避免每次重复询问
6. 若用户采用统一“指导/”目录结构，则后续实验优先从该目录读取文档与资料
7. 若同一课程存在多个实验，允许维护“实验名 -> 路径/文档”的临时映射，但不得写死为固定课程专用规则
8. 优先在项目目录内完成局部环境配置，减少对用户全局环境的影响
9. 所有截图资源必须可追溯、可复用、可插入报告

# 🎯 实验自动执行规范（AGENTS.md）

## 🌏 语言与输出规范（强约束）

1. 所有思考、分析、解释必须使用中文
2. 所有代码注释必须使用中文
3. 所有实验报告必须使用中文
4. 所有记忆文档必须使用中文
5. 输出必须结构清晰、分点说明、逻辑严谨
6. 默认采用“低上下文占用”协作方式：聊天窗口只保留必要摘要，详细过程优先沉淀到项目内文档

---

## 🎯 核心执行原则

1. 先读文档，再写代码（强制）
2. 严格按实验要求实现，不得自行发挥
3. 最小修改原则（只改必要代码）
4. 严格限制修改范围
5. 每个实验必须生成报告 + git 版本
6. 所有实验必须可回溯（版本化管理）
7. 本规范应优先保证“可复用性”，禁止把具体课程、具体实验名、具体路径硬编码进执行逻辑
8. 若编译、测试、运行环境缺失或未正确配置，必须先处理环境问题，再进入代码实现阶段
9. 每次运行成功后以及通过测试集后，必须保留输出结果截图，并写入实验报告
10. 当同一仓库内存在多个实验时，必须优先采用“实验分支隔离 + 分支内版本递增”的版本管理方式，避免不同实验之间互相污染
11. 为防止长对话导致上下文遗忘，必须为每个实验维护轻量记忆文档，用于记录当前状态、修改点、阻塞点与下一步计划
12. 记忆文档是“长期上下文索引”，不是第二份实验报告；必须简明、增量、可快速恢复上下文

---

## 📂 适用范围

本规范适用于“按实验说明完成代码修改、生成实验报告、进行 git 版本管理、维护实验记忆文档”的开发任务。

可适配：

- 操作系统实验
- 编译原理实验
- 数据库实验
- 计算机网络实验
- 其他具有“实验文档 + 代码目录 + 报告输出 + git版本管理”结构的课程实验

## ⚙️ 启动前信息收集机制（必须先完成）

当用户第一次要求开始某个实验时，如果以下信息不完整，必须先主动询问；不得在信息缺失时直接修改代码。

### 必须收集的信息

1. 实验名称
2. 实验代码目录
3. 实验文档路径
4. 报告输出路径
5. 允许修改的文件或目录范围
6. git 仓库根目录
7. 是否存在实验继承关系图或参考资料
8. 当前实验是否有固定命名规则（如 lab1_challenge2）
9. 编译、测试、运行方式
10. 是否已有可用环境，或是否允许在项目目录下补充配置环境
11. 当前仓库是“单实验仓库”还是“多实验共享仓库”
12. 若为多实验共享仓库，是否允许为每个实验创建独立分支
13. 是否允许为当前实验创建独立记忆文档
14. 若用户已有记忆文档或报告模板，路径是什么

### 推荐询问模板

请提供以下信息后再开始实验：

1. 实验名称：
2. 实验代码目录：
3. 实验文档路径：
4. 报告输出路径（若不提供，则默认生成在当前实验目录下）：
5. 允许修改的文件/目录范围：
6. git 仓库根目录：
7. 是否有继承关系图、指导文件夹或补充资料？路径是什么？
8. 是否有版本命名规则？若没有，将使用默认规则。
9. 该实验的编译命令、测试命令、运行命令分别是什么？
10. 如果环境缺失，是否允许我在项目目录下自行补充配置？若不允许，请告诉我由你负责配置。
11. 当前仓库是单实验仓库还是多实验共享仓库？
12. 若是多实验共享仓库，是否允许我为该实验建立独立分支？
13. 是否允许我为该实验创建记忆文档以节省上下文并记录进度？
14. 若已有记忆文档或报告模板，请提供路径。

### 默认约定（用户未指定时可建议）

若用户不想逐项提供，可建议采用以下统一结构：

- 指导资料统一放入：`指导/`
- 实验文档统一放入：`指导/实验文档/`
- 继承关系图统一放入：`指导/`
- 代码统一放入：`代码/`
- 每个实验报告默认放在对应实验目录下
- 运行截图与测试截图默认放在对应实验目录下的 `report_assets/` 目录中
- 记忆文档默认放在对应实验目录下的 `.agent_memory.md`
- 若同一仓库内有多个实验，则默认每个实验一个独立分支

```text
项目根目录/
├── 指导/
│   ├── 实验文档/
│   ├── 继承关系图.png

│   ├── 实验A/
│   ├── 实验B/
│   └── ...

└── AGENTS.md
```

若用户未采用该结构，也必须以“用户实际提供的路径”为准，不得自行假设。

每次开始一个新实验前，必须先整理并确认当前实验配置。

### 当前实验配置应至少包含

- 实验名称：`<experiment_name>`
- 实验目录：`<experiment_dir>`
- 实验文档：`<experiment_doc>`
- 报告路径：`<report_path>`
- 允许修改范围：`<allowed_paths>`

- 测试命令：`<test_command>`
- 运行命令：`<run_command>`
- 截图资源目录：`<report_assets_dir>`

- 记忆文档状态：`<memory_status>`

在配置不清晰前，不得直接进入代码修改阶段。

## 🧠 记忆文档机制（必须执行）

### 目标定位

记忆文档用于压缩长期上下文，帮助后续继续实验时快速恢复状态，降低聊天窗口的 token 消耗，避免因长对话导致遗忘关键修改逻辑。

### 基本原则

1. 记忆文档只记录“任务索引”和“状态摘要”，不写成长篇报告
2. 记忆文档不是第二份实验报告，禁止与报告内容大面积重复
3. 记忆文档必须支持快速阅读，默认应在极短时间内恢复当前实验上下文
4. 每次关键修改完成后必须更新记忆文档，但只更新变化项
5. 后续继续实验时，应优先读取记忆文档，再进入代码分析
6. 聊天窗口默认不重复展开记忆文档已有内容，只汇报本轮变化

### 默认路径

若用户未指定，默认使用：

```text
<experiment_dir>/.agent_memory.md
```

若用户更倾向显式文件名，也可采用：

```text
<experiment_dir>/memory.md
<experiment_dir>/<experiment_name>_memory.md
```

但同一仓库应保持一致。

### 记忆文档必须包含的最小字段

1. 当前实验名称
2. 当前实验分支
3. 当前版本
4. 当前目标
5. 本轮修改点
6. 修改原因
7. 影响范围
8. 当前结果
9. 未解决问题
10. 下一步计划
11. 报告路径
12. 截图目录

### 推荐结构

```md
# 实验记忆

## 基本状态
- 实验：
- 分支：
- 当前版本：
- 文档状态：
- 环境状态：
- 报告路径：
- 截图目录：

## 当前目标
- 

## 最近一次修改
- 修改点：
- 原因：
- 涉及文件：
- 结果：

## 当前阻塞 / 风险
- 

## 下一步
- 

## 历史简记
- [v1] ...
- [v2] ...
```

### 写作要求

1. 每条记录应简短、明确、可索引
2. 优先使用短句和项目符号
3. 不放大段代码
4. 不重复插入截图
5. 不复制实验报告正文
6. 若某信息已经稳定且长期不变，不应在每轮反复重写
7. 历史简记只保留关键节点，不保留细碎过程噪声

### 更新触发时机

以下场景必须更新记忆文档：

1. 第一次启动实验并完成配置确认后
2. 完成 initial 保存后
3. 完成一轮有效代码修改后
4. 编译/运行/测试结果出现新结论后
5. 出现新的阻塞点或风险后
6. 保存新版本后
7. 切换版本、回退版本、从历史版本继续开发后

### 读取规则

1. 当用户说“继续某个实验”时，必须优先读取该实验记忆文档
2. 若记忆文档存在，则先以其中内容恢复当前上下文，再决定是否重读报告
3. 若记忆文档不存在，则在第一次有效执行后创建
4. 若记忆文档内容明显过长，应主动压缩为摘要形式

## 🌿 分支与版本管理总规则（必须遵守）

### 基本原则

1. 分支是“实验隔离单元”，不是目录别名
2. 版本是“当前实验分支内的演进记录”，不是跨实验共享状态
3. 当同一仓库内有多个实验时，必须优先采用“一个实验一个分支”的管理方式
4. `main`（或用户指定主分支）只作为公共稳定基线，不直接承担多个实验的日常迭代
5. 默认不将实验分支自动合并回主分支，除非用户明确要求
6. 版本切换必须优先确认当前所在实验分支，禁止在错误分支上直接切版本
7. 若仓库仅用于单个实验，且用户明确不需要分支隔离，则允许退化为“单分支 + tag”模式

### 默认分支策略

若用户未指定规则，则默认采用：

- 主分支：`main`
- 实验分支：与实验名称同名，例如 `<experiment_name>`
- `main` 保存内容：
  - 初始工程
  - 公共配置
  - 通用脚本
  - 通用文档
  - AGENTS.md / skill / 通用报告模板等
- 实验分支保存内容：
  - 该实验相关代码修改
  - 该实验报告
  - 该实验截图资源
  - 该实验版本提交与标签
  - 该实验记忆文档

### 切换实验与切换版本的区别

1. 切换实验：本质是切换到该实验对应分支
2. 切换版本：本质是在当前实验分支内切换到指定 tag / commit / 历史版本
3. 禁止把“切换实验”和“切换版本”混为一个动作处理

## 🛠️ 环境检查与配置规则（必须执行）

### 环境检查触发条件

在开始编码前，必须先检查以下内容是否可用：

1. 编译工具链是否可用
2. 测试命令是否可执行
3. 运行命令是否可执行
4. 依赖是否齐全
5. 项目要求的容器、虚拟环境、解释器、编译器、模拟器、构建工具是否齐全
6. 当前目录结构是否满足实验运行要求

### 当环境未配置好时的处理规则

若发现编译、测试、运行环境未配置好，必须暂停代码实现，先输出以下内容并向用户确认：

1. 当前缺失的环境项
2. 每个环境项的用途
3. 建议的配置方式
4. 是否需要在项目目录下新增配置文件、脚本、依赖目录或容器配置
5. 这些配置会修改哪些文件
6. 这些修改是否只用于环境搭建而非实验逻辑实现

### 必须向用户确认的标准话术要求

当环境缺失时，必须明确询问用户：

当前编译/测试/运行环境未配置完成。
是否允许我在项目目录下自行补充环境配置？

我计划配置的内容包括：

- 缺失环境 1：用途说明
- 缺失环境 2：用途说明
- 缺失环境 3：用途说明

预计会新增或修改以下内容：

- 文件/目录 A
- 文件/目录 B
- 文件/目录 C

这些改动仅用于让项目可以正常编译、测试和运行，不涉及实验业务逻辑修改。
请确认是否允许我执行。

### 若允许自行配置

若用户明确允许，则可以在项目目录下补充必要环境配置，但必须遵守：

1. 优先使用项目内局部配置，不污染全局环境
2. 优先新增独立配置文件，避免破坏原项目
3. 所有环境相关新增内容必须可说明、可回退、可追踪
4. 环境配置也必须写入实验报告
5. 环境配置完成后，必须重新验证编译、测试、运行流程
6. 环境配置结论必须同步更新到记忆文档

### 若不允许自行配置

若用户不允许，则必须：

1. 明确指出缺失项
2. 告知用户需要其协助完成哪些操作
3. 为用户提供可直接执行的详细步骤
4. 在用户完成前，不得继续代码实现

### 需要用户协助时的输出要求

若需要用户配置环境，必须明确写出：

1. 要安装什么
2. 去哪里安装
3. 在哪个目录执行
4. 执行哪些命令
5. 配置完成后如何验证
6. 验证成功后需要把什么输出反馈回来

### 环境配置说明必须包含的内容

若执行或建议环境配置，必须说明：

1. 环境名称
2. 作用
3. 为什么当前实验需要它
4. 安装或配置位置
5. 对项目的影响范围
6. 是否需要用户权限、网络、系统依赖或容器支持

------

## 🚫 代码修改限制（强约束）

只能修改“用户明确允许修改”的文件或目录。

### 执行规则

1. 若用户已明确给出允许修改范围，则严格以用户要求为准
2. 若用户未给出允许修改范围，必须先询问
3. 禁止修改实验目录之外的无关文件
4. 禁止擅自扩大修改范围
5. 若发现实现目标必须修改范围外文件，必须先向用户说明并征求确认
6. 环境配置文件与实验逻辑文件应区分说明，禁止将环境改动混入业务改动而不告知用户
7. 在多实验共享仓库中，除非用户明确要求，否则不得顺手修改其他实验目录
8. 若当前实验采用独立分支管理，仍然必须遵守允许修改范围，分支隔离不等于允许越界修改

------

## 🔄 实验执行流程（统一）

当用户输入：

- 开始某个实验
- 实现某个实验
- 修改某个实验
- 继续某个实验

必须执行以下流程：

### 第一步：确认实验配置

1. 收集或确认实验名称
2. 定位实验目录
3. 定位实验文档
4. 确认允许修改范围
5. 确认报告输出路径
6. 确认 git 仓库根目录
7. 确认编译、测试、运行命令
8. 确认截图保存目录
9. 确认环境配置权限
10. 确认仓库是单实验还是多实验共享
11. 确认是否允许创建实验分支
12. 确认主分支名与实验分支名
13. 确认记忆文档路径或创建策略

### 第二步：读取记忆文档

1. 若记忆文档存在，优先读取
2. 提取当前目标、最近修改点、当前阻塞点与下一步
3. 若记忆文档不存在，则在完成首次有效执行后创建
4. 若记忆文档过长，则先压缩摘要后再继续

### 第三步：检查环境

1. 检查编译环境
2. 检查测试环境
3. 检查运行环境
4. 若环境缺失，则先向用户确认是否允许在项目目录下自行配置
5. 若需要用户协助，则输出详细操作步骤并等待用户完成
6. 环境就绪后，再继续后续流程

### 第四步：阅读文档

1. 阅读实验文档
2. 提炼实验目标
3. 总结实验要求
4. 若存在继承关系图、参考代码或补充资料，也应一并阅读

### 第五步：检查分支策略

1. 判断当前实验是否应使用独立分支
2. 若为多实验共享仓库，默认必须使用独立实验分支
3. 若实验分支不存在，则从主分支创建
4. 若实验分支已存在，则切换到该分支
5. 若用户明确要求不使用分支隔离，必须先提示风险后再执行

### 第六步：版本初始化检查

1. 检查当前是否已切换到目标实验分支
2. 检查该实验是否已存在 initial 版本
3. 若不存在，则先保存 initial
4. 若已存在，则跳过
5. initial 的创建、查询与切换均以“当前实验分支”为前提
6. initial 创建结果必须同步写入记忆文档

### 第七步：代码分析

1. 分析需要修改的文件
2. 分析需要修改的函数
3. 给出修改思路
4. 在未分析清楚前，不得直接改代码

### 第八步：实现代码

1. 仅在允许范围内修改
2. 保持最小改动
3. 保持原有代码风格一致
4. 关键逻辑补充中文注释

### 第九步：编译、运行与测试

1. 执行编译
2. 执行运行
3. 执行测试
4. 每次运行成功后保存输出截图
5. 通过测试集后保存输出截图
6. 整理输出结果，用于写入实验报告
7. 将最新结论压缩写入记忆文档

### 第十步：生成实验报告

1. 按规范输出实验报告
2. 插入运行成功截图
3. 插入测试通过截图
4. 保存到约定路径
5. 在记忆文档中记录报告路径与当前报告状态

### 第十一步：保存版本

1. 在当前实验分支内自动递增版本号
2. 提交 git commit
3. 创建对应 tag
4. 若远程存在对应实验分支，则推送该分支与 tag
5. 默认不自动合并回主分支
6. 保存版本后必须更新记忆文档中的当前版本与历史简记


5. 当前实验标签记录
6. 当前实验记忆文档

## 🧪 initial 版本规则（必须执行）

### initial 的作用

`initial` 仅表示“当前实验分支第一次开始实现前的起始快照”，不是整个仓库所有实验共享的公共版本。

### initial 命名规则

若用户未提供规则，则默认采用：

```bash

git commit -m "chore: save <experiment_name> initial"
git tag <experiment_name>_initial
```

### 规则说明

1. 每个实验只应创建一次 initial
2. 若 `<experiment_name>_initial` 已存在，则跳过
3. 禁止重复覆盖 initial
4. initial 必须在目标实验分支内创建
5. 若当前不在目标实验分支，不得创建 initial

## 🏷️ 版本保存规则（自动递增）

### 默认版本命名

若用户未提供规则，则默认：

```bash
git add -A
git commit -m "feat: complete <experiment_name> vN"
git tag <experiment_name>_vN
```

### 版本号获取规则

1. 必须先确认当前位于目标实验分支
2. 使用 git tag 查询当前实验已有版本：

```bash
git tag | grep "^<experiment_name>_v[0-9]\+$"
```

1. 提取最大版本号
2. 在其基础上 +1
3. 禁止覆盖已有版本
4. 版本号只对当前实验生效，不与其他实验共用序号
5. 若用户采用自定义版本命名规则，则按用户规则执行，但仍需保证“实验间互不混淆”

若存在远程仓库，默认执行：

```bash
git push origin <experiment_branch>
git push origin <experiment_name>_vN
```

或：

```bash
git push origin <experiment_branch> --tags
```

具体以项目实际规范为准。

### 与记忆文档同步规则

每次保存新版本后，必须同步更新：

- 当前版本
- 本轮修改点摘要
- 当前结果
- 下一步
- 历史简记

## 🔁 版本切换机制

当用户输入：

- 切换到某个版本
- 回到某个版本
- checkout 到某个版本

必须先判断用户是要：

1. 切换实验
2. 切换实验内版本
3. 仅查看某个历史版本
4. 从某个历史版本继续开发

### 标准处理规则

#### 情况一：用户说“切到某个实验”

执行逻辑：

1. 确认目标实验名
2. 定位该实验分支
3. 检查分支是否存在
4. 切换到该实验分支
5. 读取对应记忆文档恢复状态

默认命令：

```bash
git checkout <experiment_branch>
```

#### 情况二：用户说“切到某实验的某个版本”

执行逻辑：

1. 先切到该实验分支
2. 再检查目标版本是否存在
3. 再切换到对应 tag / commit
4. 切换后更新记忆文档状态

默认命令：

```bash
git checkout <experiment_branch>
git checkout <target_version>
```

#### 情况三：用户仅要查看历史版本

允许进入 detached HEAD 状态，但必须告知。

#### 情况四：用户要从历史版本继续开发

必须提示先新建分支，例如：

```bash
git checkout <target_version>
git checkout -b <target_version>_dev
```

### 必须提示

若当前处于 detached HEAD 状态，必须提示：

当前处于 detached HEAD 状态，
如需继续开发，建议新建分支后再修改，例如：

```bash
git checkout -b <target_version>_dev
```

### 严格限制

1. 禁止在未确认实验分支的情况下直接切版本
2. 禁止把某实验的版本误用于另一实验分支
3. 若用户指定的版本存在歧义，必须先澄清是“实验分支”还是“版本标签”
4. 若仓库是多实验共享仓库，默认优先保护实验隔离，不应为了省事直接在主分支上回退

------

## 🔀 主分支同步规则（仅在用户明确要求时执行）

默认情况下，实验分支完成后：

1. 提交到当前实验分支
2. 推送当前实验分支
3. 保留该实验 tag
4. 不自动 merge 回主分支

仅当用户明确提出以下诉求时，才考虑同步主分支：

- 把当前实验结果合并回主分支
- 用当前实验结果更新公共基线
- 将实验成果整理入主线统一版本

执行前必须先说明：

1. 将影响主分支
2. 可能改变后续新实验的起始基线
3. 是否会对其他实验分支造成潜在基线差异
4. 合并后建议重新标记主分支稳定版本

## 📊 输出规范（低 Token 模式，必须遵守）

### 总原则

1. 聊天窗口默认采用“摘要模式”
2. 首次启动实验时允许完整确认一次
3. 后续继续实验时只输出变化项，不重复整套状态
4. 关键事件才进入“展开模式”
5. 详细留痕优先写入记忆文档和实验报告，不在聊天窗口长篇重复

### 输出模式

#### 模式一：首次完整模式

仅在以下场景使用：

1. 第一次启动某实验
2. 当前实验配置明显不完整
3. 用户明确要求详细列出
4. 仓库/分支/环境策略首次建立时

首次完整模式应输出：

1. 实验名称
2. 实验目录
3. 文档位置
4. 允许修改范围
5. 编译/测试/运行命令
6. 仓库类型
7. 主分支名
8. 实验分支名
9. 记忆文档路径
10. 当前环境是否可用
11. 当前目标

#### 模式二：默认摘要模式

除关键事件外，后续对话默认使用摘要模式，只输出：

1. 当前实验
2. 当前分支
3. 当前版本
4. 本轮目标
5. 本轮修改点或本轮阻塞点
6. 当前结果
7. 是否已更新记忆文档 / 报告 / 版本

#### 模式三：关键事件展开模式

仅在以下场景展开详细输出：

1. 环境缺失
2. 需要用户授权
3. 需要越界修改
4. 创建/切换分支
5. 切换版本 / 回退版本
6. 编译失败 / 运行失败 / 测试失败
7. 实验完成
8. 用户明确要求详细过程

### 聊天窗口禁止事项

1. 不得在每轮重复完整配置清单
2. 不得在每轮复述完整报告内容
3. 不得在每轮复制记忆文档全文
4. 不得把历史所有修改记录都重新讲一遍
5. 不得为了“显得完整”而持续输出高冗余内容

### 文档优先级规则

1. 当前协作摘要：放聊天窗口
2. 长期状态索引：放记忆文档
3. 正式完整交付：放实验报告
4. 版本留痕：放 git commit / tag / 分支历史

14. 默认采用“首次完整、后续增量、关键事件展开”的沟通方式
15. 优先把长期状态沉淀到记忆文档，而不是沉淀到聊天窗口
16. 若记忆文档已能覆盖当前上下文，则聊天中不重复展开历史细节
